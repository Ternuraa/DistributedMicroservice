NFR 

Нефункциональные требования к системе Harbor: целевые показатели по задержкам, пропускной способности, доступности и гарантиям консистентности


1. Latency (задержка)

| Создание бронирования User + Listing + Booking | ≤ 500 ms | 2–3 синхронных gRPC-вызова; оптимизация за счёт параллельных вызовов User и Listing где возможно получение деталей листинга (каталога) | ≤ 100 ms | Чтение из БД/кэша; критично для UX поиска |
| Отправка уведомления (приём в очередь) | ≤ 50 ms | Только приём сообщения в брокер; доставка email/push — асинхронно |
| Получение отзывов по листингу | ≤ 150 ms | Чтение + агрегация; можно кэшировать |

Обоснование:
 пользователь не должен ждать дольше ~0.5 s при бронировании; каталог и отзывы — быстрые ответы для отзывчивого интерфейса.


Throughput (пропускная способность)

| Listing Service (поиск, детали) | 1000+ RPS | Самый нагружаемый; кэш, индексы, при необходимости шардирование |
| Booking Service (создание/обновление брони) | 200+ RPS | Ограничено транзакциями и проверками доступности |
| User Service (GetUserStatus и т.п.) | 500+ RPS | Лёгкие запросы; кэш по user_id |
| Notification (приём событий в очередь) | 500+ msg/s | Очередь принимает пики; обработка — по возможности воркеров |
| Review Service (чтение отзывов) | 500+ RPS | Чтение доминирует; кэш агрегатов по listing_id |

Обоснование:
Пики нагрузки — в выходные и при акциях; целевые значения позволяют планировать масштабирование (горизонтальное масштабирование сервисов и очередей).


3. Availability (доступность)

| Доступность системы в целом | ≥ 99.5% | Не более ~3.5 ч простоя в месяц в сумме |
| Критичные сервисы (Booking, Listing, User) | ≥ 99.9% | Реплики, health checks, автоматический перезапуск; отказ одной ноды не роняет сервис |
| Notification Service | ≥ 99% | Допустимы кратковременные задержки доставки уведомлений; очередь сохраняет сообщения |

Обоснование: 
Бронирование и каталог — ключевые сценарии; их недоступность напрямую влияет на выручку. Уведомления важны, но их задержка не должна считаться полным отказом системы

4. Consistency (гарантии консистентности)

| Создание брони + доступность дат | **Strong consistency** в рамках одного листинга | Транзакция в Booking Service: резерв дат и создание записи reservation в одной БД; при необходимости пессимистичные блокировки или оптимистичная конкуренция с проверкой версии |
| Цена в бронировании | **Strong consistency** на момент создания | Цена берётся синхронно из Listing при создании брони и сохраняется в reservation; последующие изменения цены в каталоге не меняют уже созданные брони |
| Статус брони после оплаты | **Eventual consistency** между Payment и Booking допустима на секунды | SetReservationPaid вызывается после успешного платежа; при временной недоступности Booking — retry и идемпотентность по payment_id |
| Рейтинги и отзывы на витрине | **Eventual consistency** допустима | Агрегаты (средний рейтинг, число отзывов) пересчитываются после записи отзыва; кэш с TTL 1–5 мин приемлем |
| Уведомления | **At-least-once delivery** | Очередь с подтверждением обработки; идемпотентность обработки в Notification, чтобы повтор доставки не дублировал push/email |

Итог:
* Strong: бронирование и доступность дат, фиксация цены в брони, критичные проверки (UserStatus, наличие листинга).

* Eventual: рейтинги, агрегаты отзывов, статус «оплачено» при retry; уведомления — at-least-once
