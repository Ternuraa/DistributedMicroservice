Sync vs Async 

1. Синхронная коммуникация (gRPC)

Где используется:

BookingService → UserService.GetUserStatus — перед созданием брони нужно сразу знать, верифицирован ли гость и не забанен

BookingService → ListingService.GetListingDetails — нужны актуальная цена и доступность для расчета суммы и проверки дат в одном запросе

ReviewService → BookingService.GetReservation — при отправке отзыва нужно сразу проверить, что бронирование завершено 

Listing / BFF → ReviewService.GetReviewsForListing, GetUserRating — отображение рейтингов на витрине и в профиле требует немедленного ответа

Обоснование:

* Ответ нужен в рамках одного use case: без него нельзя продолжить операцию (создать бронь, показать страницу)
* Требуется строгая консистентность: решение принимается на основе актуальных данных (цена, статус пользователя)
* Задержка в 50–200 ms допустима цепочка из 2–3 синхронных вызовов укладывается в приемлемое время


2. Асинхронная коммуникация (очереди сообщений / события)

Где используется:
Любой сервис → NotificationService.SendSystemNotification — уведомления (email, push) после бронирования, оплаты, нового отзыва, нового сообщения в чате

Обоснование:
* Ответ от Notification не нужен для завершения основной операции: бронь уже создана, платеж уже прошел
* Уведомления можно доставить с задержкой в несколько секунд без потери корректности бизнес-логики
* Сбой Notification (временная недоступность, перегрузка) не должен блокировать создание брони или приём платежа — слабая связанность
* Пиковые нагрузки (много бронирований) сглаживаются очередью: сообщения обрабатываются по мере возможности — устойчивость к нагрузке

Реализация:
публикация события в брокер Notification Service подписан на эти события и отправляет уведомления. Либо вызов gRPC в fire-and-forget режиме с повторными попытками и очередью внутри Notification
